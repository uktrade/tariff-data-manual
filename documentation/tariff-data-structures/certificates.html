Certificates

  • Introduction to certificates

Certificates are associated to measures via measure conditions, this term refers to the certificates, licenses, and non-paper conditions that are stored in certificate tables and are required to bring goods through customs.

Introduction to certificates

What are certificates?

The certificates tables in TARIC-based applications store all of the certificates, licences and other conditions that are referenced via measure conditions in order to make the import (or export) of a good conditional on the provision of some entity.

The sole purpose of certificates in this system is to associate them to measures via measure conditions – on their own, they no value and are not exposed to traders via any other means.

This entity is usually a physical or digital document of some description (e.g. a license or official certificate), but it may also be a non-paper condition - the term “certificate” to label the table is definitely more constrictive than the actual subject matter.

A good example of the ‘non-document’ condition is “Y037 - Goods exempted from the prohibition – biathlon equipment’ where the condition references a property of the good being imported, and not an accompanying document.

These oddities are typically linked to measures using the measure_condition_code “Y”. Please read the instruction guide on measure conditions for a detailed insight.

What certificates are not

  • A certificate is not a representation of the actual document itself
  • Our database stores only a certificate code (4 digits) and a description of that code
  • There is no instruction on where to find that document (if the certificate represents an actual document), there is no link to a URL nor guidance on how to complete the certificate
  • This could be construed as a flaw in the public-facing aspect of the certification model.
  • There is a feeling of ‘so what’ when confronted with the need for a certificate – the interaction may favour traders that know certification inside out and the regulations that underpin it, but for entryists, the concept can be bewildering
  • DIT may consider extending the model to allow more information than just a code and basic description to be stored against the certificate, bearing in mind that the majority of such documents or other entities would likely be owned by Defra or on occasion HMRC

Structure of certificates & certificate types

  • Much like footnotes and additional codes, certificates are made up of a type code and then a specific code, which, when concatenated, form the unique code of the certificate.
  • For example, the footnote C656 is constructed of a type code © and a specific certificate code (656) – the entire certificate is known as C656.
  • Certificate type codes are always 1 digit in length and can be any alphanumeric character (upper case). Historically the alphabetical certificate type codes have been used for EU-wide certificates: the numeric certificate type codes are used for national measures.
  • Certificate codes typically use numeric IDs, running from 001 to 999 can be reused across multiple certificate type codes. While the schema also allows for alphabetical certificate codes, these are rarely used: there are 11 instances of certificate codes using alpha instead of numeric digits.
  • Certificate codes must be 3 digits in length.

Who owns certificates?

  • It is anticipated that there will be numerous owners for certificates across government: Defra, the RPA, ECJU, Trade Remedies Authority for example
  • In the lead-up to the previous three abortive No Deal EU Exits, the question of migrating certification to a standard that differed from the EU’s was not mentioned – all certification would have remained exactly as-is, for two reasons:
    • this was the best way to prevent additional ‘accidental’ friction at the border
    • the expertise and knowledge in government to make informed changes to certification was not there
  • After the implementation period, the expectation is that standards will diverge to a degree and therefore certification will also diverge, however for Jan 1st 2021, the expectation (that needs to be tested) is that the EU certification will be carried forward as-is.
  • Please see the document on data migration for more information.
