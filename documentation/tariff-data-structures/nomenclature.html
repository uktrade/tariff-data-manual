Background

Combined Nomenclature, Harmonised System - Structure

HS - Harmonised System
  • The first six digits of the code numbers of the tariff or TARIC have been regulated by the World Customs Organization (WCO) in the form of the HS nomenclature world-wide since 1988.
  • HS is currently used in 176 countries. A revision takes place regularly, roughly every 5 years (so far, 1996, 2002, 2007, 2012, 2017). Next change anticipated in 2022.
CN - Combined Nomenclature
  • The “Combined Nomenclature” is published in the Official Journal of the European Community in October each year and specifies the first eight digits of the Tariff/Taric, as well as third country tariff rates for the following year. This eight-digit nomenclature can not be changed over a period of one year and applies both in the EU and the Associated States (BG, RO, TR, etc.).
TARIC - Tarif Intégré Communautaire / Tarif intégré des Communautés européennes
  • Taric, the tariff rate of the European Communities, is the same as the CN in the first eight digits. There is, therefore, a uniform marking for customs purposes and statistics. The Taric is subdivided however by the Taric sub-positions (from the ninth place) more deeply.
  • As the combined nomenclature is not sufficiently detailed to support the community tariff and commercial legislation, the 8-digit codes can be broken down into 10-digit codes called “TARIC codes”.
  • These codes are created at any time, according to the legislative needs.

Harmonised system sections

  • The Harmonised System is categorised into sections and chapters
  • The section is a logical grouping that is not reflected in the nomenclature.
  • Each chapter, which is a subdivision of a section, is fully self-contained within the logical section.
  • Sections are represented by Roman numerals (1 to 21 / I to XXI)

Concept definition & terminology 2

  • The UK refers to the full list of products in the UK Tariff as the Goods Classification – this is synonymous with the broad name for the full list of products: goods nomenclature or just nomenclature.
  • In the EU, this is referred to under multiple names: often the EU refers to the Tariff as the “Combined Nomenclature” (CN), though officially this refers only to the first 8 digits of the code.
  • Within the broader body, the individual elements referring to specific products are called commodity codes.
  • The commodity code is what is entered into the SAD (C88) form – the Single Administrative Document that acts as the import declaration form – see ringed box to the right.
  • There is a total of 24,504 commodity codes (as of 1st December 2019): of these 15,887 are declarable end-lines (see definition later).

There are three key concepts to understand in this section. All of these have a significant impact on the measures that can be defined and managed by Tariff Managers:

  • commodity codes
  • product line suffixes
  • indents

Commodity codes, product line suffixes and indents in practice

  • The schematic to the right illustrates a ‘hierarchy’ starting from the very first commodity code in the Tariff (0100 00 00 00) down to the last end-line under the 1st major HS heading branch (0101 90 00 00), i.e. Live horses, asses, mules and hinnies
  • This shows the commodity codes, split into segments for ease of reading, the product line suffix and the indent, as well as the current description of the commodity code.
  • The physical indent on the description matches the indent field.
  • Hence, Horses is a ‘child’ of ‘Live horses, asses, mules and hinnies’ and ‘Pure-bred breeding animals’ is a child of ‘Horses’ – however these relationships are inferred, not explicit.
  • You will notice that there are two zeroes at the start of the tree – this is a slightly bewildering treatment from the EU, but we need to work with it: the HS chapter and the HS heading both use an indent of 0. This will not be of huge immediate interest, as these codes will not change until the lead-up to January 2022.

If one were to try and print out the whole commodity tree (c. 24,500 lines), you would:

  • select the commodity codes with current validity
  • sort them by commodity code and then product line suffix
  • indent them using the indent field

Declarability and measure assignment

  • A SAD form requires that the trader enter the most detailed description possible for the good that is being traded, not something vague like ‘Horses’
  • There are two rules that define whether or not a good is declarable, i.e. whether or not it can be entered onto a SAD form:
  • first, a good must have a product line suffix (PLS) of “80”: anything else cannot be traded – these commodity codes are in the database purely as structural fillers
  • second, a good must be an end-line aka a leaf. i.e. it must not have any children.
  • In the example to the left, the declarable commodities are those with a against them
  • Although 0101 29 00 00, 01010 00 00 00 and 0101 00 00 00 all have a PLS of “80”, they all have child codes underneath them, therefore cannot be declared.

  • In the example above (red marquee), it is interesting to note (and this is key in understanding where a product line suffix other than “80” is used), that there are two items with the commodity code 0101 21 00 00.
  • One has a PLS of 10, the other has a PLS of 80.
  • This is common and is likely to be a treatment we need to get used to in the UK for future change.

Smashing preconceptions on hierarchical inheritance

  • This is undoubtedly an unusually complex tree, however it does illustrate well a concept that needs to be understood, to help the Tariff management team to grasp the idea of inheritance, which is critical in measure management.
  • The 10-digit commodity code is divided into 5 pairs of digits.
  • While it is alluring to consider that these two digit pairs provide enough information to work out the hierarchy, this is coincidental, though, certainly at the top of the hierarchy, frequently true.

There are two additional concepts that are useful to know, but play a far smaller part in the management of goods classification. Origins and successors are intended to create an audit trail when commodity codes change

Origins

Origins

  • An origin defines the commodity code from which a new commodity code was derived.
  • When a commodity code is split into two, the ‘origin’ of the two (or more) new commodity codes is the parent commodity code.

  • In the example above, a successor will have been created for codes 2001 90 70 10 and 2001 90 70 90 of 2001 90 70 00.
  • It is mandatory that at least one origin is created for every new commodity code
  • However, this data is of zero value to the Tariff Application

Successors

  • An successor defines the commodity code into which an existing commodity code has been absorbed.
  • It is not mandatory for a successor code to be created, though they may be created.

Residual codes

  • The sweet peppers example can also be used to exemplify residual codes.
  • A residual code is the code usually labelled ’Other’ which is created when a code is split into a specific and remainder part.
  • In this example 2001 90 70 00 is split into 2 child codes.
    • The specific code is assigned 10 as the final 2 digits.
    • The residual code is assigned 90 as the final 2 digits.
    • If in future more child codes as added to the parent as siblings to 10 and 90, then these are assigned 20, 30, 40 etc as the final 2 digits, while space remains.

  • If a code is split on the last digit, rather than the last 2 digits, it is usually split into 1 and 9, for example:

  • There are more than 6,500 residual codes in the database

Key business rules


#
Rule
Notes

NIG1
The validity period of the goods nomenclature must not overlap any other goods nomenclature with the same goods code. Two nomenclatures overlap if they have the same goods code and a time overlap in their validity periods.
Mandatory – priority high

Commodity codes are uniquely identified by their SID value. There cannot be more than one entity with the same SID. There can however be more than one entity with the same compound key of commodity code and product line suffix, provided the multiple items do not coexist concurrently. The use case here is that a commodity code has previously been terminated, and then reinstituted at a later date with the same commodity code and product line suffix. This rule states that there must be no crossover in the dates of these entities that shar the same commodity code and product line suffix.

NIG2
The validity period of the goods nomenclature must be within (inclusive) the validity period of the product line above in the hierarchy.
Mandatory – priority high

If terminating a commodity code, you must have already done the same to the child commodity codes.

NIG3
For lines having a product line suffix equal to “80”, the validity period of the goods nomenclature must span the validity period of the line(s) below in the hierarchy.
Mandatory – priority high

This is essentially the same as NIG2

NIG4
The start date of the goods nomenclature must be less than or equal to the end date.
Mandatory – priority high

As per rule description


Goods nomenclature origin

NIG5
When creating a goods nomenclature code, at least one “derived from”‘ code must be created.
This rule is only applicable to update extractions.
Mandatory – priority high

The origin should be the code from which the commodity is derived (i.e. its parent) if the use case id the creation of a new commodity code from a parent, e.g. for splitting a commodity code into two.

If the use case concerns moving chunks of or single commodity codes from one place in the hierarchy to another , then the requirement is that the origin should be the identifier of the code which used to represent the commodity in question

NIG7
The “derived from” code, if entered, must be a goods nomenclature which exists and is applicable the day before the start date of the new code entered.
Mandatory – priority high

as per rule


Goods nomenclature successor

NIG10
The “absorbed by” code, if entered, must be a goods nomenclature which exists and is applicable the day after the closing date.
Mandatory – priority high

as per rule, however there are no rules which state that successors must be entered, though it remains a good idea to do so


Goods nomenclature indent

NIG11
At least one indent record is mandatory. The start date of the first indentation must be equal to the start date of the nomenclature. No two associated indentations may have the same start date. The start date must be less than or equal to the end date of the nomenclature.
Mandatory – priority high

as per rule

as per rule

Footnote association

NIG18
Footnotes with a footnote type for which the application type = “CN footnotes” must be linked to CN lines (all codes up to 8 digits).
Mandatory – priority low

This is a peculiarity that should be consigned to history – we should imply not use footnotes of this type in the future. The proposal is for all future footnotes to belong to the new footnote type of “FC”, which will assignable to any commodity code, regardless of ‘depth’, so that daft rules such as this do not need to be run - see the notes on new footnote types in the footnotes manual.

NIG19
Footnotes with a footnote type for which the application type = “TARIC footnotes” can be associated at any level.
Mandatory – priority low

The “FC” type, mentioned above would be of a type with the application code “Taric footnotes”, therefore can be assigned to any commodity code, regardless of its depth.

NIG20
Footnotes that are referenced must exist.
Mandatory – priority high

User must only be able to select existing footnotes

NIG21

Mandatory – priority medium

NIG22
The period of the association with a footnote must be within (inclusive) the validity period of the nomenclature.
Mandatory – priority medium

as per rule, but in most cases there is no need to end-date a footnote to commodity code association.

NIG23

Mandatory – priority medium

as per rule, but in most cases there is no need to end-date a footnote to commodity code association.

NIG24

Mandatory – priority medium

as per rule, but in most cases there is no need to end-date a footnote to commodity code association.

NIG37

Optional

Relation with measures
